Interfaces on Demand

Shenzhe Zhu · May 9, 2026

Claude Design made one trend feel very concrete: agents are starting to generate interfaces, not just text. The product pairs chat with a canvas, then lets the user iterate on a working design with messages and inline comments. Export is also part of the loop: the same design can become a standalone HTML file, a Canva handoff, or a handoff to Claude Code.

The important shift is not only "AI makes mockups faster." It is that the interface can be generated at the moment of need. A task does not always need a fixed product screen that was designed months ago. It may need a temporary editor, a comparison board, a set of buttons for choosing an export format, or a small canvas for arranging assets.

HTML for Humans

Markdown is still useful. It is compact, easy for agents to produce, and good for plain explanations. But the final surface is often for a human, not for another model. HTML is better when the answer needs layout, hierarchy, buttons, forms, SVG, tabs, sliders, or export actions.

Simon Willison's note on the "unreasonable effectiveness of HTML" makes the same point from another angle: instead of returning a wall of Markdown, an agent can return a small self-contained page. Thariq Shihipar's examples show code reviews, design systems, prototypes, diagrams, research explainers, and custom editors as HTML artifacts. These are not just prettier documents. They are easier to scan, react to, and reuse.

Agentic-Native Platforms

A future agentic-native platform may not have one fixed interface. It may have a stable core product, plus many task-specific surfaces generated on the fly. One user needs a dashboard. Another needs a Canva-like asset picker. Another needs a prompt tuner, a feature-flag editor, or a visual plan. The same app can become many small apps.

This is especially relevant for collaboration products. In a system like Kordi, the agent should not only write messages or produce documents. It should create the interface that lets a user decide, arrange, compare, export, and give feedback. The user stays in the loop, but the loop becomes more visual and much tighter.

The Product Implication

"Anything can be HTML" does not mean every response should become a webpage. Plain text is still right for many cases. The point is that HTML gives agents a native way to make human-facing, interactive output. When the task involves choice, comparison, design, review, editing, or export, the interface should probably be generated directly.

The product lesson is simple: agents should optimize not only for correct answers, but for the right surface. Sometimes that surface is Markdown. Increasingly, it may be a small piece of HTML made exactly for the user and the task in front of them.
