On-Demand Agent-Native Interaction Surfaces

Shenzhe Zhu · May 9, 2026

A small moment in Claude Design feels less like an isolated feature and more like an early signal. Agent-native applications will increasingly generate the interaction surface a user needs in the moment, not only return text inside a chat window.

Claude Design makes this trend concrete because it pairs chat with a canvas, inline comments, direct edits, and export paths to HTML, Canva, or Claude Code. But the larger point is not HTML itself. HTML is only the easiest visible example right now. The real shift is that an agent-native app can generate interaction at the moment of need.

A task does not always need a fixed product screen designed months ago. It may need a temporary editor, a comparison board, a set of controls, or a small canvas for arranging assets. This matters because user experience is not only about visual polish. It is also about interaction performance: how quickly a user can understand the state, make a choice, give feedback, and move to the next step.

The way I would frame it: an agent-native platform should not treat UI as a permanent container. It should treat UI as a material the system can reshape around the user's current intention.

HTML as One Surface

Markdown is still useful. It is compact, easy for agents to produce, and good for plain explanations. But the final surface is often for a human, not for another model. HTML is useful because it can carry layout, hierarchy, buttons, forms, SVG, tabs, sliders, and export actions in one familiar runtime.

Simon Willison's note on the "unreasonable effectiveness of HTML" makes the same point from another angle: instead of returning a wall of Markdown, an agent can return a small self-contained page. Thariq Shihipar's examples show code reviews, design systems, prototypes, diagrams, research explainers, and custom editors as HTML artifacts. These are not just prettier documents. They show how one agent answer can become a more usable interaction surface.

Agent-Native Interaction

A future agent-native platform may not have one fixed interface. It may have a stable core product, plus many task-specific surfaces generated on the fly. One user needs a dashboard. Another needs a Canva-like asset picker. Another needs a prompt tuner, a feature-flag editor, or a visual plan. The same app can become many small apps.

This is especially relevant for collaboration products. In a system like Kordi, the agent should not only write messages or produce documents. It should create the interface that lets a user decide, arrange, compare, export, and give feedback. The user stays in the loop, but the loop becomes more visual and much tighter.

Some product functions feel natural here: temporary control panels for parameter tuning, comparison boards for choosing between generated options, editable previews before handoff, visual review surfaces for comments, task-specific dashboards, and export panels that know the user's next tool. If a generated surface becomes useful repeatedly, the product can promote it into a saved workflow. That is how an agent-native app can evolve without forcing every workflow into the same fixed screen.

The Product Implication

"Anything can be HTML" does not mean every response should become a webpage. Plain text is still right for many cases, and future surfaces may also be canvas, spreadsheets, forms, simulations, voice, or mixed interfaces. The important question is not "should this be HTML?" The better question is "what interaction does this task need right now?"

The product lesson is simple: agents should optimize not only for correct answers, but for the right interaction surface. Sometimes that surface is Markdown. Sometimes it is HTML. Increasingly, it may be a small, generated workspace made exactly for the user and the task in front of them.

My prediction is that this will stop feeling like a special design-tool feature. In agent-native applications, on-demand interaction surfaces may become a normal product primitive: generated when the task needs them, discarded when the moment passes, and saved only when they become a repeated workflow.
